Controlador de versiones

El control de versiones es un sistema que registra los cambios en un archivo o conjunto de archivos a lo largo del tiempo para que luego se pueda recuperar una versión específica de dicho archivo o proyecto. En este sentido es posible ver quién cambió por última vez algo que podría haber causado un problema, ver quién causó el problema y cuándo y mucho más. Existen diferentes sistemas que permiten tener un control de versiones: sistemas de control de versiones locales, centralizado y distribuido.

El sistema de control de versiones locales es un método popular y sencillo. Consiste en copiar los archivos a otro directorio, sin embargo puede traer problemas, ya que es fácil olvidar en qué directorio se encuentra y guardarlo accidentalmente en el archivo equivocado o sobrescribir un archivo que no desea.

Con el sistema de control de versiones locales no se previno la colaboración de diferentes personas en un mismo proyecto, y como solución a este problema se desarrolló un sistema de control de versiones centralizado (CVCS). Estos sistemas tienen un servidor que almacena todas las versiones de los archivos donde varios clientes pueden descargar dichos archivos desde esa ubicación central.

Este sistema funcionaba muy bien, sin embargo tenía el gran inconveniente de que si el servidor central dejaba de funcionar, los clientes no podían guardar o actualizar los cambios hechos en el archivo donde se encontraba colaborando. Así mismo, si el disco duro donde se guardan todos los archivos se daña y no se ha creado una copia de estos archivos, éstos se perderían totalmente.

Al contrario del sistema anterior, el sistema de control de versiones distribuido permite no solo que los clientes descarguen la última versión del archivo, sino que replicar todo el repositorio, de manera que éste está disponible para recuperarse en caso de que el servidor falle. En otras palabras, si el servidor deja de funcionar, cualquier repositorio disponible en el cliente se puede copiar al servidor para restaurarlo.

Estos sistemas también ofrecen la posibilidad de trabajar en diferentes proyectos al mismo tiempo y colaborativamente.

En mi opinión, el sistema de control de versiones local es una buena práctica como desarrollador independiente, es decir, que no necesita colaborar con otros desarrolladores. Sin embargo, cuando se trata de un proyecto colaborativo, sería muy conveniente utilizar un sistema distribuido, ya que no se corre el riesgo de perder los archivos y éstos se encuentran altamente disponibles
